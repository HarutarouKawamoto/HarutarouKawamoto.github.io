# Blog Creation Skill — ブログ作成ワークフロー

このスキルは、ブログ記事の作成を「指示」と「確認」だけで完結させるためのワークフロー定義です。
人間は「テーマを伝える」「完成物を確認する」だけでOKです。

---

## ワークフロー全体像

```
[Step 1] テーマ受け取り・意図の明確化
     ↓
[Step 2] トレンド・競合分析
     ↓
[Step 3] SEOキーワード設計
     ↓
[Step 4] 記事構成（アウトライン）作成
     ↓
[Step 5] 本文執筆
     ↓
[Step 6] フロントマター（メタデータ）生成
     ↓
[Step 7] 自動レビュー（agents並行実行）
     ↓
[Step 8] 修正・最終化
     ↓
[Step 9] ファイル保存・ステータス更新
```

---

## Step 1: テーマ受け取り・意図の明確化

ユーザーからテーマを受け取ったら、以下の情報が揃っているか確認する。
**情報が不足している場合のみ**質問する（揃っていれば即座にStep 2へ）。

必須情報チェックリスト:
- [ ] 記事テーマ / キーワード（必須）
- [ ] ターゲット読者（不明なら「一般的な技術読者」と仮定してよい）
- [ ] 想定文字数（不明なら1500〜2500文字で執筆する）
- [ ] トーン（不明なら「わかりやすく親しみやすい」で執筆する）

**質問は一度にまとめて行い、2回以上の往復は禁止。**

---

## Step 2: トレンド・競合分析

以下の分析を実行する:

```
分析項目:
1. 同テーマの上位記事で取り上げられているサブトピック
2. 読者がよく抱く疑問・痛み（読者の検索意図）
3. まだカバーされていない独自の切り口（差別化ポイント）
```

**出力形式:**
```markdown
## トレンド分析サマリー
- よく取り上げられるサブトピック: ...
- 読者の主な疑問: ...
- 差別化できる切り口: ...
```

---

## Step 3: SEOキーワード設計

### メインキーワード選定ルール
- 検索ボリュームと競合難易度のバランスを考慮
- タイトルに必ずメインキーワードを含める
- URLスラグはメインキーワードのkebab-case

### キーワード配置ルール
| 場所 | キーワード種別 | 必須/推奨 |
|------|--------------|----------|
| タイトル（h1） | メインキーワード | 必須 |
| 最初の100文字以内 | メインキーワード | 必須 |
| メタディスクリプション | メインキーワード + LSI | 必須 |
| h2見出し（1〜2箇所） | 関連キーワード | 推奨 |
| alt属性 | 関連キーワード | 推奨 |

### SEOスコアリング基準（seo-reviewerに引き継ぐ）
```
タイトル文字数（30-60文字）          : 20点
メタディスクリプション（120-160文字）: 20点
キーワード密度（1-3%）               : 20点
見出し構造（h1→h2→h3の階層）        : 15点
内部リンク（1記事に2本以上）         : 15点
画像alt属性（全画像に設定）          : 10点
                              合計: 100点
```

---

## Step 4: 記事構成（アウトライン）作成

### 推奨構成テンプレート

```markdown
# [タイトル] — メインキーワードを含む（50文字以内）

## 導入（100〜150文字）
- 読者の課題・疑問を共感的に提示
- この記事で解決できることを明示

## [メインセクション1] — h2
### [サブセクション1-1] — h3
### [サブセクション1-2] — h3

## [メインセクション2] — h2
### [サブセクション2-1] — h3

## [メインセクション3] — h2

## まとめ
- 記事のキーポイントを箇条書きで3点
- 次のアクション（CTAまたは関連記事への誘導）
```

**構成を作成したらユーザーに提示し、承認を得てからStep 5へ進む。**
承認なしに本文執筆を開始してはいけない。

---

## Step 5: 本文執筆

### 執筆ルール

**文体・トーン:**
- 一文は60文字以内を目安にする（日本語）
- 専門用語には初出時に説明を付ける
- 受動態より能動態を優先する
- 「〜である」「〜だ」よりも「〜です」「〜ます」を基本とする（指定がない場合）

**構造ルール:**
- 大きな段落（200文字超）は分割する
- リストは3〜7項目にまとめる
- コードブロックには必ず言語を指定する（例: ` ```typescript `）
- 画像は必ずalt属性を記述する

**禁止事項:**
- ❌ 「〜という形になります」（冗長）→「〜です」に変換
- ❌ 「〜させていただきます」（過剰敬語）→「〜します」に変換
- ❌ 根拠のない数値・統計の引用
- ❌ 特定製品・サービスへの過度な誘導

## Step 6: フロントマター生成

本文完成後、CLAUDE.mdで定義された必須フィールドを含むフロントマターを自動生成する:

```yaml
---
title: "記事タイトル（50文字以内）"
description: "メタディスクリプション（120〜160文字、メインキーワード含む）"
date: "YYYY-MM-DD"  # 今日の日付
tags: ["タグ1", "タグ2", "タグ3"]
author: "Claude"
slug: "url-friendly-slug"
status: "review"  # 初期値はreview（published禁止）
reading_time: "N分"  # 本文文字数 ÷ 400文字/分で計算
seo:
  canonical: "https://example.com/blog/{slug}"
  og_image: "/images/og/{slug}.png"
  keywords: ["キーワード1", "キーワード2"]
---
```

---

## Step 7: 自動レビュー（agents並行実行）

本文とフロントマターが完成したら、以下の3エージェントを**並行して**起動する。

```
並行実行:
├── seo-reviewer     → SEO要件チェック（100点満点でスコアリング）
├── security-reviewer → XSS・サニタイズ・依存関係チェック
└── label-checker    → UIテキストハードコーディングチェック
```

各エージェントの詳細は `.claude/agents/` 配下のファイルを参照。

**レビュー結果をユーザーに提示するまで次のステップへ進んではいけない。**

---

## Step 8: 修正・最終化

エージェントのレビュー結果を受け取り:

1. **Critical（致命的）** な問題は自動修正する
2. **Warning（警告）** はユーザーに提示して判断を仰ぐ
3. **Info（情報）** はレポートとして記録する

修正後、品質ゲート（CLAUDE.mdのセクション5）をすべて通過しているか確認する。

---

## Step 9: ファイル保存・ステータス更新

```
保存先: posts/YYYY-MM-DD-{slug}.md
ステータス更新:
  - 品質ゲート通過 → "review"（ユーザーが最終確認後に"published"へ変更）
  - 品質ゲート未通過 → "draft"（要修正）
```

**ユーザーへの最終報告フォーマット:**
```
✅ 記事生成完了

📄 ファイル: posts/YYYY-MM-DD-{slug}.md
📊 SEOスコア: {score}/100
🔒 セキュリティ: 問題なし / {N}件の警告
🏷️ ラベルチェック: 問題なし / {N}件の警告
📝 文字数: {count}文字 / 読了時間: {N}分

次のステップ: 記事を確認し、問題なければ status を "published" に変更してください。
```
