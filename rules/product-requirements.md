# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**HarutarouKawamoto Portfolio** - エンジニアポートフォリオ＆技術ブログサイト

### プロダクトコンセプト
- **第一印象で「誰か」を伝える**: ヒーローセクションで一言でアイデンティティを訴求し、採用担当者・エンジニア友人に即座に人物像を伝える
- **スキルと成果物で技術力を証明する**: カテゴリ分け・熟練度可視化・成果物カードにより、言葉ではなく実績でエンジニアとしての価値を示す
- **継続的な技術発信で信頼を積み上げる**: 週1目標のブログ更新により、学習・思考・実践のプロセスを発信し続ける

### プロダクトビジョン
エンジニア友人と就活採用担当者の両方に刺さるポートフォリオサイトを構築する。
技術スタック・成果物・ブログ記事を通じて「何ができるか」と「どういう人か」を同時に伝え、就職活動における差別化を実現する。
日英バイリンガル対応により、国内外問わず自分をアピールできる場を作る。
継続的な技術発信によって、スキルアップの過程そのものを資産として蓄積していく。

### 目的
- 就職・転職活動における自己PR媒体の構築
- 技術スキルと成果物の一元管理・公開
- 週次のブログ発信による学習の言語化と習慣化
- 国内外の採用担当者・エンジニアとのつながりの創出

## ターゲットユーザー

### プライマリーペルソナ1: 就活採用担当者（鈴木さん、35歳、HR担当）
- 採用候補者のポートフォリオを短時間でスクリーニングする

- 使用技術・成果物・最新記事を確認して「最近何してるか」を把握したい
- **現在の課題**: 「SNSだけだと技術の深さがわからない」
- **期待する解決策**: Skills と Blog から最近の学習領域と技術的な思考を確認できること
- **典型的なワークフロー**: SNSリンク → Home → Skills（技術スタック確認）→ Blog（最新記事）

## 機能要件

### コア機能(MVP)

#### 1. Homeページ（ヒーローセクション）

**ユーザーストーリー**:
採用担当者として、サイトを開いた瞬間に「このエンジニアは誰か」を把握するために、明確なキャッチコピーとナビゲーションが欲しい

**受け入れ条件**:
- [ ] キャッチコピー（一言で自分を表す文言）が3秒以内に視認できる
- [ ] 氏名・職種・GitHubリンクがファーストビューに表示される
- [ ] 各セクション（About/Skills/Products/Blog/Contact）へのナビリンクが機能する
- [ ] ダーク/ライトモード切り替えボタンが設置されている
- [ ] 日英切り替えボタンが設置されている

**優先度**: P0（必須）

---

#### 2. Aboutページ（自己紹介）

**ユーザーストーリー**:
採用担当者として、候補者の人柄・価値観・経歴を把握するために、読みやすい自己紹介ページが欲しい

**受け入れ条件**:
- [ ] 氏名・自己紹介文・価値観・経歴が掲載されている
- [ ] 日本語・英語それぞれで適切な内容が表示される
- [ ] 3分以内に全体を読み通せる量（800〜1200字相当）に収める

**優先度**: P0（必須）

---

#### 3. Skillsページ（技術スタック）

**ユーザーストーリー**:
エンジニア友人として、使用技術の全体像と熟練度を把握するために、カテゴリ別・熟練度付きのスキル一覧が欲しい

**受け入れ条件**:
- [ ] カテゴリ（言語 / フレームワーク / ツール）でグルーピングされている
- [ ] 言語カテゴリのみ熟練度を5段階で視覚的に表示する
- [ ] フレームワーク・ツールは熟練度なしのシンプルな一覧表示
- [ ] スキルの追加・変更がデータファイル（JSON/TS）の編集のみで完結する

**優先度**: P0（必須）

---

#### 4. Productsページ（成果物一覧）

**ユーザーストーリー**:
採用担当者として、実装能力を確認するために、成果物の概要・技術スタック・リンクを一覧で確認したい

**受け入れ条件**:
- [ ] 成果物がカード形式で表示される
- [ ] 各カードに「開発中」「完成」のステータスタグが表示される
- [ ] 使用技術タグが複数付与できる
- [ ] GitHubリンクボタン・デモリンクボタンがそれぞれ独立して設置できる（任意項目）
- [ ] 成果物の追加がデータファイルの編集のみで完結する

**優先度**: P0（必須）

---

#### 5. Blogページ（技術発信）

**ユーザーストーリー**:
採用担当者として、候補者の技術的思考・学習意欲を確認するために、カテゴリ・タグで整理された記事一覧を読みたい

**受け入れ条件**:
- [ ] 記事一覧がカード形式で表示される（タイトル・カテゴリ・日付・概要）
- [ ] 記事一覧はページネーション表示とし、1ページあたり10件を表示する
- [ ] 次のページへの遷移リンクが機能する
- [ ] カテゴリ（Featured / New / Learn / Enjoy / Real）でフィルタリングできる（カテゴリ定義は `rules/glossary.md` 参照）
- [ ] タグでフィルタリングできる
- [ ] 記事はMarkdownで執筆し、ビルド時にHTMLへ変換される
- [ ] 日本語記事と英語記事が対応付けられ（`translationSlug`）、言語切り替えで遷移できる
- [ ] 対応する翻訳記事が存在しない場合、言語切り替えボタンは非表示またはブログ一覧へ誘導する（フォールバック）
- [ ] `blog-creation` スキルで生成した記事がそのまま掲載できる

**優先度**: P0（必須）

---

#### 6. Contactページ（お問い合わせ）

**ユーザーストーリー**:
採用担当者として、候補者に連絡を取るために、簡単に送信できるお問い合わせフォームが欲しい

**受け入れ条件**:
- [ ] フォーム項目: お名前 / メールアドレス / 電話番号（任意）/ 内容 / 送信ボタン
- [ ] Formspree経由で送信が完了する
- [ ] 送信成功・失敗のフィードバックがUIに表示される
- [ ] スパム対策としてhoneypotフィールドを設置する

**優先度**: P0（必須）

---

#### 7. 多言語対応（i18n）

**ユーザーストーリー**:
海外の採用担当者として、英語でポートフォリオ内容を確認するために、全ページが英語表示に切り替えられる機能が欲しい

**受け入れ条件**:
- [ ] ナビバーに言語切り替えボタン（JA / EN）が設置されている
- [ ] ボタン1つで全ページの表示言語が日本語 ↔ 英語に切り替わる
- [ ] URL構造が `/ja/...` と `/en/...` で分離されている
- [ ] 初回アクセス時にブラウザの言語設定（`navigator.language`）を自動判定して表示言語を決定する
- [ ] 言語設定がlocalStorageに保存され、再訪問時に維持される

**優先度**: P0（必須）

---

#### 8. ダーク/ライトモード

**ユーザーストーリー**:
サイト閲覧者として、自分の好みの表示環境で閲覧するために、ダーク/ライト切り替えが欲しい

**受け入れ条件**:
- [ ] ナビバーにダーク/ライト切り替えボタンが設置されている
- [ ] システムのカラースキーム（`prefers-color-scheme`）を初期値として使用する
- [ ] 設定がlocalStorageに保存され、再訪問時に維持される
- [ ] Tailwind CSSの`dark:`クラスで全コンポーネントが対応している

**優先度**: P0（必須）

各ページ・記事に最適化されたOGP画像が自動生成される機能

**優先度**: P1（重要）

**着手条件**: MVPリリースと同時に設定（プライバシーポリシーの記載が必要）

**優先度**: P2（できれば）
**着手条件**: Blog記事が50件以上になった段階で検討

## 非機能要件

### パフォーマンス
- Lighthouse Performance スコア: 90点以上
- First Contentful Paint (FCP): 1.5秒以内
- Largest Contentful Paint (LCP): 2.5秒以内
- Cumulative Layout Shift (CLS): 0.1以下
- Interaction to Next Paint (INP): 200ms以内
- Time to First Byte (TTFB): 600ms以内（GitHub Pages CDN経由）
- 画像は WebP 形式または `<picture>` タグで最適化する
- 詳細な測定方法は `rules/architecture.md` のパフォーマンス要件を参照

### ユーザビリティ
- ナビゲーションから任意のページへ2クリック以内で到達できる
- モバイル（375px〜）・タブレット（768px〜）・デスクトップ（1280px〜）でレイアウトが崩れない
- 言語・テーマ切り替えが1クリックで完了する

### 信頼性
- GitHub Pages でのホスティングによる高可用性
- ビルドエラーを GitHub Actions CI でブロックし、壊れたページを本番に出さない

### セキュリティ
- お問い合わせフォームにhoneypot・Formspreeのスパムフィルタを使用する
- 外部スクリプトはSRI（Subresource Integrity）ハッシュで検証する
- 個人情報（メールアドレス等）をソースコードにハードコードしない

### アクセシビリティ
- セマンティックHTMLを使用し、スクリーンリーダーで主要コンテンツが読み上げられる
- 画像に適切な`alt`テキストを設定する
- キーボードのみで全操作が完結できる

### スケーラビリティ
- Blog記事が100件を超えても `npm run build` が60秒以内に完了する
- Blog記事が100件を超えても GitHub Actions のデプロイ（ビルド + GitHub Pages反映）が3分以内に完了する
- スキル・成果物のデータをTypeScriptの型付きデータファイルで管理し、型安全な拡張ができる

## スコープ外

明示的にスコープ外とする項目:
- バックエンドサーバー・データベース（静的サイトとして完結）
- ユーザー認証・ログイン機能
- コメント機能（Blogへのコメント投稿）
- 有料コンテンツ・決済機能
- 英語以外の多言語対応（日英の2言語のみ）
- CMS管理画面（記事はMarkdownファイルで直接管理）
